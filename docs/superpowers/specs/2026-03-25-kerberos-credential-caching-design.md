# Kerberos Credential Caching System Design

**Date:** 2026-03-25  
**Status:** Approved  
**Author:** GitHub Copilot

## Overview

Implement a credential caching system for Kerberos authentication to eliminate repetitive AWS Secrets Manager API calls. Credentials will be fetched once from AWS, stored securely in local files, and automatically loaded on shell startup.

## Problem Statement

Currently, `party-db-auth` fetches Kerberos credentials from AWS Secrets Manager on every invocation. This:

- Slows down authentication (network round-trip to AWS)
- Requires active AWS SSO session for routine authentication
- Creates unnecessary API calls when credentials rarely change

## Goals

1. **First-run credential caching** - Fetch from AWS once, save locally
2. **Fast authentication** - Use cached credentials without AWS calls
3. **Manual refresh mechanism** - Update credentials when they change
4. **Security** - Store credentials with proper file permissions
5. **Simplicity** - Follow existing patterns (like `.azure_devops_pat`)

## Non-Goals

- Automatic credential expiration/rotation
- Encrypted credential storage (WSL local storage is trusted)
- Multi-user credential sharing

## Architecture

### File Structure

**New Files:**

```
~/.party/
├── sit-credentials  (chmod 600)
└── qas-credentials  (chmod 600)
```

**Modified Files:**

- `utils/risk_rangers_env.sh` - Sources credential files on shell startup
- `utils/party-auths.sh` - Modified `party-db-auth` function

### Credential File Format

Each `~/.party/{env}-credentials` file contains:

```bash
# Auto-generated by party-db-auth - do not edit manually
# Last updated: 2026-03-25 08:30:15 NZDT

# Environment-specific credentials (new format)
export KERBEROS_PRINCIPAL_SIT="svc_account@CORP.BANK.SIT.KIWIBANK.INTERNAL"
export KERBEROS_PASSWORD_SIT="<secure_password>"
export SDT_SERVER_SIT="SITCFC-Listener.corp.bank.sit.kiwibank.internal,50530"
export SDT_DATABASE_SIT="ServicelayerDataTier"
export REALM_SIT="CORP.BANK.SIT.KIWIBANK.INTERNAL"

# Backward compatibility (old format) - deprecated
export KERBEROS_PRINCIPAL="$KERBEROS_PRINCIPAL_SIT"
export KERBEROS_REALM="$REALM_SIT"
export SDT_SERVER="$SDT_SERVER_SIT"
export SDT_DATABASE="$SDT_DATABASE_SIT"
export KERBEROS_ENV="SIT"
```

**Note:** Both old and new variable formats are exported for backward compatibility. The environment-specific format (`_SIT`, `_QAS`) allows multiple environments to coexist. The non-suffixed variables provide compatibility with existing scripts.

### Component Design

#### 1. risk_rangers_env.sh Changes

Add credential file sourcing after existing environment setup:

```bash
# Load cached Kerberos credentials if available
if [ -f "$HOME/.party/sit-credentials" ]; then
    # Validate file permissions before sourcing
    if [ "$(stat -c %a "$HOME/.party/sit-credentials" 2>/dev/null || stat -f %A "$HOME/.party/sit-credentials" 2>/dev/null)" = "600" ]; then
        source "$HOME/.party/sit-credentials"
    else
        echo -e "${RED}[WARNING]${NC} Skipping $HOME/.party/sit-credentials - insecure permissions (expected 600)" >&2
    fi
fi

if [ -f "$HOME/.party/qas-credentials" ]; then
    # Validate file permissions before sourcing
    if [ "$(stat -c %a "$HOME/.party/qas-credentials" 2>/dev/null || stat -f %A "$HOME/.party/qas-credentials" 2>/dev/null)" = "600" ]; then
        source "$HOME/.party/qas-credentials"
    else
        echo -e "${RED}[WARNING]${NC} Skipping $HOME/.party/qas-credentials - insecure permissions (expected 600)" >&2
    fi
fi
```

**Behavior:**

- Validates file permissions (must be 600) before sourcing
- Warns if permissions are insecure and skips sourcing
- Silently succeeds if files don't exist (user hasn't authenticated yet)
- Credentials available as environment variables for all shells
- Uses portable `stat` command (works on both Linux and macOS)

**Cache Check Logic (Shell-Agnostic):**

```bash
# Determine if we need to fetch from AWS
local need_fetch=false
local env_upper=$(echo "$env" | tr '[:lower:]' '[:upper:]')

# Check for --refresh flag
if [[ "${2:-}" == "--refresh" ]]; then
    need_fetch=true
    echo -e "${YELLOW}[INFO]${NC} Refreshing credentials from AWS..."
fi

# Check if credentials exist in environment (shell-agnostic approach)
# Using eval instead of ${!var} for bash/zsh compatibility
local principal_check
local password_check
eval "principal_check=\${KERBEROS_PRINCIPAL_${env_upper}:-}"
eval "password_check=\${KERBEROS_PASSWORD_${env_upper}:-}"

if [ -z "$principal_check" ] || [ -z "$password_check" ]; then
    need_fetch=true
    echo -e "${YELLOW}[INFO]${NC} Credentials not cached. Fetching from AWS..."
fi
```

**Shell Compatibility Note:** The `eval` approach works in both bash and zsh. Alternative methods like `${!var}` (bash) or `${(P)var}` (zsh) are shell-specific and avoided for portability.

**Cache Save Logic (Atomic with Secure Permissions):**

```bash
# Create ~/.party directory if it doesn't exist
mkdir -p "$HOME/.party"
chmod 700 "$HOME/.party"

# Write credentials to cache file with atomic permissions
local cache_file="$HOME/.party/${env}-credentials"
local temp_file="${cache_file}.tmp.$$"

# Create temp file with secure permissions from the start
(
    umask 077  # Ensure file created with 600 permissions
    cat > "$temp_file" <<EOF
# Auto-generated by party-db-auth - do not edit manually
# Last updated: $(date '+%Y-%m-%d %H:%M:%S %Z')

# Environment-specific credentials (new format)
export KERBEROS_PRINCIPAL_${env_upper}="${full_principal}"
export KERBEROS_PASSWORD_${env_upper}="${password}"
export SDT_SERVER_${env_upper}="${server}"
export SDT_DATABASE_${env_upper}="${database}"
export REALM_${env_upper}="${realm}"

# Backward compatibility (old format) - deprecated
export KERBEROS_PRINCIPAL="\${KERBEROS_PRINCIPAL_${env_upper}}"
export KERBEROS_REALM="\${REALM_${env_upper}}"
export SDT_SERVER="\${SDT_SERVER_${env_upper}}"
export SDT_DATABASE="\${SDT_DATABASE_${env_upper}}"
export KERBEROS_ENV="${env_upper}"
EOF
)

# Verify temp file was created successfully
if [ ! -f "$temp_file" ]; then
    echo -e "${RED}[ERROR]${NC} Failed to write credential cache"
    return 1
fi

# Double-check permissions (paranoid validation)
chmod 600 "$temp_file"

# Atomically replace old cache with new cache
mv -f "$temp_file" "$cache_file"

# Source the file to update current shell
source "$cache_file"

echo -e "${GREEN}[INFO]${NC} Credentials cached to ${cache_file}"
```

**Security Notes:**

- `umask 077` ensures file created with 600 permissions (no race window)
- Temp file with $$ (process ID) prevents concurrent write collisions
- Atomic `mv -f` replaces old cache safely
- Double chmod 600 as paranoid validation

### API Changes and Backward Compatibility

**⚠️ BREAKING CHANGE:** Environment variable naming has changed to support multiple environments simultaneously.

**Old Interface (Deprecated but Still Supported):**

```bash
export KERBEROS_PRINCIPAL="user@CORP.BANK.SIT.KIWIBANK.INTERNAL"
export KERBEROS_REALM="CORP.BANK.SIT.KIWIBANK.INTERNAL"
export KERBEROS_ENV="SIT"
export SDT_SERVER="SITCFC-Listener.corp.bank.sit.kiwibank.internal,50530"
export SDT_DATABASE="ServicelayerDataTier"
```

**New Interface (Primary):**

```bash
export KERBEROS_PRINCIPAL_SIT="user@CORP.BANK.SIT.KIWIBANK.INTERNAL"
export KERBEROS_PASSWORD_SIT="<secure_password>"
export REALM_SIT="CORP.BANK.SIT.KIWIBANK.INTERNAL"
export SDT_SERVER_SIT="SITCFC-Listener.corp.bank.sit.kiwibank.internal,50530"
export SDT_DATABASE_SIT="ServicelayerDataTier"

export KERBEROS_PRINCIPAL_QAS="user@CORP.BANK.QAS.KIWIBANK.INTERNAL"
export KERBEROS_PASSWORD_QAS="<secure_password>"
export REALM_QAS="CORP.BANK.QAS.KIWIBANK.INTERNAL"
export SDT_SERVER_QAS="SQLCFC-QAS.corp.bank.sit.kiwibank.internal,50532"
export SDT_DATABASE_QAS="ServicelayerDataTier"
```

**Compatibility Strategy:**

Both old and new variable formats are exported by the cache files. The unsuffixed variables (old format) represent the most recently authenticated environment, providing backward compatibility with existing scripts that reference `$KERBEROS_PRINCIPAL`.

**Migration for Scripts:**

Scripts should migrate to environment-specific variables:

```bash
# OLD (still works, but ambiguous)
kinit $KERBEROS_PRINCIPAL

# NEW (explicit and supports multiple environments)
kinit $KERBEROS_PRINCIPAL_SIT
kinit $KERBEROS_PRINCIPAL_QAS
```

**Deprecation Timeline:**

- Phase 1 (Current): Both formats supported, old format uses most recent auth
- Phase 2 (Future): Deprecation warning when using old format
- Phase 3 (Long-term): Remove old format support (requires scripts to migrate)

## Error Handling

### Missing Cache on First Run

- **Detection:** Environment variables not set
- **Action:** Automatically fetch from AWS and cache
- **Message:** `[INFO] Credentials not cached. Fetching from AWS...`

### AWS Fetch Failures

- **Detection:** AWS CLI returns non-zero exit code
- **Action:** Display error, **preserve existing cache** (don't overwrite)
- **Behavior:** If cached credentials exist, they remain valid and usable for kinit
- **Message:**
  ```
  [ERROR] Failed to fetch credentials from AWS Secrets Manager
  [HINT] Make sure you're logged in: aws sso login --profile ciptooling-prod-party
  [INFO] Using existing cached credentials (if available)
  ```
- **Return:** Exit code 1 if fetch fails AND no cache exists; proceed with cached credentials otherwise

**AWS SSO Expiration Scenarios:**

1. **Cache exists + AWS SSO expired + `--refresh` flag:**
   - AWS fetch fails → Display error
   - Keep existing cache intact
   - Proceed to kinit with cached credentials
   - User can still authenticate if cached credentials are valid

2. **Cache exists + AWS SSO expired + no refresh flag:**
   - Skip AWS fetch (using cache)
   - Proceed normally with cached credentials
   - No AWS SSO required for normal authentication

3. **No cache + AWS SSO expired:**
   - Cannot fetch credentials → Error and exit
   - User must login: `aws sso login --profile ciptooling-prod-party`

### Permission Errors

- **Prevention:** Always set directory to 700, files to 600
- **Detection:** mkdir/chmod/cat failures
- **Action:** Display error and abort
- **Message:** `[ERROR] Failed to create/update credential cache`

### Invalid Environment

- **Detection:** Environment not in [sit, qas]
- **Action:** Display usage and exit
- **Message:**
  ```
  Error: Invalid environment '$env'. Use 'sit' or 'qas'
  Usage: party-db-auth [sit|qas] [--refresh]
  ```

## Security Considerations

### File Permissions

- **Directory:** `~/.party/` set to 700 (owner read/write/execute only)
- **Files:** `~/.party/*-credentials` set to 600 (owner read/write only)
- **Enforcement:** Permissions set every time credentials are written

### Storage Location

- Credentials stored in user's home directory (not in repository)
- WSL environment provides local machine security
- No cross-user access possible with 600 permissions

### Credential Handling

- Passwords never logged or echoed to terminal
- AWS fetches use stderr redirection for error messages only
- Credential variables prefixed with `KERBEROS_` for clarity

### Git Protection

- `~/.party/` stored in user home, not in repository
- Should add `.party/` to global gitignore as best practice
- Repository `.gitignore` already excludes home directory files

- Remove cache loading from `risk_rangers_env.sh`
- Function reverts to fetching on every call (like before)
